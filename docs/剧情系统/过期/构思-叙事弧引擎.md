> **文档类型**：构思/设计草案  
> **创建日期**：2026-05-25  
> **状态**：待评审

---

## 📋 概览

**系统名称**：叙事弧引擎（Narrative Arc Engine）

**核心定位**：在现有 `StoryManager`（主线5章）+ `QuestManager`（传统任务追踪）+ `EventBus`（事件总线）之上，新增一个**通用的叙事状态机引擎**，用于驱动：
- 隐藏剧情线（玩家自己发现）
- 多分支多结局的复杂支线
- NPC记忆与因果关系
- 世界永久改变
- 连锁反应（一个故事的结局触发另一个故事）

**设计哲学**：
> 轮子不限制创意，轮子释放创意。引擎提供"状态机+条件+动作"的通用骨架，策划只需写配置文件就能创造任意复杂的剧情。

---

## 🏗️ 架构设计

### 与现有系统的关系

```
┌─────────────────────────────────────────────────────┐
│                  叙事弧引擎（新增）                    │
│  NarrativeArcEngine → 编排所有现有系统                │
│  WorldStateTracker  → 世界记忆（标记/数值/NPC记忆）    │
│  ConditionEvaluator → 条件判断（桥接所有系统）         │
│  ActionExecutor     → 动作执行（调用所有系统API）      │
├─────────────────────────────────────────────────────┤
│                  现有系统（保持不变）                   │
│  StoryManager    — 主线5章剧情推进                    │
│  QuestManager    — 传统任务追踪                       │
│  EventBus        — 全局事件总线                       │
│  ReputationManager — 声望系统                        │
│  StorageBagManager — 背包物品                        │
│  MarketFluctuationManager — 经济系统                 │
│  TimeManager     — 时间系统                          │
└─────────────────────────────────────────────────────┘
```

**关键决策**：叙事弧引擎**不替代**任何现有系统，而是在上层**编排**它们。通过 EventBus 监听世界变化，通过 ActionExecutor 调用现有系统的 API。

### 四大核心模块

| 模块 | 职责 | 类比 |
|------|------|------|
| **WorldStateTracker** | 世界的"记忆"——标记、数值、NPC记忆、时间戳 | 游戏的全局黑板 |
| **ConditionEvaluator** | 判断"什么时候触发/转换"——支持40+种条件类型 | if语句的通用化 |
| **ActionExecutor** | "选择之后世界发生什么"——桥接所有现有系统 | 函数调用的通用化 |
| **NarrativeArcEngine** | 状态机核心——管理弧的生命周期和阶段转换 | 导演/编排器 |

## 🎯 核心概念

### "叙事弧" vs "任务"

| 传统任务 | 叙事弧 |
|---------|--------|
| 玩家主动接取 | 世界状态自然触发（可完全隐藏） |
| 线性（目标→完成） | 有向图（任意分支+合流） |
| 做完就完了 | 一个弧的结局可以触发另一个弧 |
| 超时无后果 | **不作为本身就是一种选择**（timeout → 默认结局） |
| 与世界无关 | 每个选择都改变世界状态 |
| 只有"完成/失败" | 多种结局，每种都有独特后果 |

### 叙事弧的生命周期

```
休眠(Dormant) → 激活(Active) → 阶段推进(Stage Transition) → 完成(Completed)
     ↑                              ↓
     └── 条件不满足时保持休眠 ←── 连锁触发新弧
```

### 阶段转换的四种方式

1. **player_choice** — 玩家主动选择（对话选项/行动）
2. **auto** — 条件满足自动转换（状态变化触发）
3. **timeout** — 超时默认转换（不作为的后果）
4. **event** — 特定事件触发转换

## 📦 数据结构设计

### StoryArcDefinition（叙事弧定义）

```
ArcID          — 唯一标识
Title          — 标题（为空则对玩家不可见=隐藏弧）
Description    — 描述
Trigger        — 触发定义
  ├── VisibleToPlayer — 是否对玩家可见
  └── Conditions[]    — 触发条件列表（AND逻辑）
Stages[]       — 阶段列表
  ├── StageID         — 阶段ID
  ├── Description     — 阶段描述
  ├── Transitions[]   — 转换列表
  │   ├── TargetStageID    — 目标阶段
  │   ├── TransitionType   — 转换类型（player_choice/auto/timeout/event）
  │   ├── Conditions[]     — 转换条件
  │   ├── TimeoutDays      — 超时天数（timeout类型专用）
  │   ├── ChoiceText       — 选项文本（player_choice类型专用）
  │   └── UnavailableHint  — 条件不满足时的提示
  ├── OnEnterActions[]     — 进入阶段时执行的动作
  ├── OnExitActions[]      — 离开阶段时执行的动作
  ├── IsEnding             — 是否是终结阶段
  └── EndingTag            — 结局标签（用于后续弧的条件判断）
```

### ArcCondition（条件定义）

支持的条件类型：

| 类别 | 条件类型 | 说明 |
|------|---------|------|
| 世界状态 | `flag` / `flag_not` | 标记存在/不存在 |
| 世界状态 | `value_gte` / `value_lte` / `value_eq` | 数值比较 |
| NPC记忆 | `npc_memory` / `npc_memory_not` | NPC是否有某条记忆 |
| 叙事弧 | `arc_completed` / `arc_ending` / `arc_not_completed` | 弧完成状态 |
| 物品 | `has_item` / `item_count_gte` | 持有物品 |
| 声望 | `reputation_gte` / `reputation_lte` | 声望值 |
| 修炼 | `realm_gte` | 境界等级 |
| 位置 | `at_location` | 当前位置 |
| 任务 | `quest_state` | 任务状态 |
| 关系 | `relationship_gte` | 好感度 |
| 时间 | `days_since_gte` / `time_of_day` / `month_gte` | 时间条件 |
| 组合 | `and` / `or` / `not` | 逻辑组合 |
| 随机 | `random` | 概率触发 |

### ArcAction（动作定义）

支持的动作类型：

| 类别 | 动作类型 | 说明 |
|------|---------|------|
| 世界状态 | `set_flag` / `set_value` / `modify_value` | 修改世界状态 |
| NPC记忆 | `add_npc_memory` | 给NPC添加记忆标签 |
| 物品 | `give_item` / `take_item` | 给予/拿走物品 |
| 货币 | `modify_currency` | 修改灵石 |
| 声望 | `modify_reputation` | 修改门派声望 |
| 关系 | `modify_relationship` | 修改好感度 |
| 任务 | `start_quest` / `complete_quest` / `fail_quest` | 操作任务 |
| 叙事弧 | `start_arc` / `advance_arc` | 启动/推进另一个弧（连锁！） |
| 阵营 | `modify_alignment` | 修改正邪权重 |
| 通知 | `notification` / `dialogue` | 发送通知/触发对话 |
| 市场 | `market_event` | 触发市场事件 |
| 时间 | `record_timestamp` | 记录时间戳 |

## 🔗 与现有系统的集成方式

### 初始化顺序

```
GameManager.InitializeGameplaySystems() 中：
  1. WorldStateTracker.Instance.Initialize()  ← 在所有系统之前
  2. ... 其他系统初始化 ...
  3. NarrativeArcEngine.Instance.Initialize() ← 在所有系统之后
```

### 存档集成

```
SaveManager.Save() 中新增：
  data.WorldState = WorldStateTracker.Instance.GetSaveData()
  data.NarrativeArcs = NarrativeArcEngine.Instance.GetSaveData()
```

### 事件监听

NarrativeArcEngine 通过 EventBus 监听以下事件来触发检测：
- `TimeAdvancedEvent`（时间推进）
- `LocationChangedEvent`（位置变化）
- `WorldFlagChangedEvent`（标记变化）
- `WorldValueChangedEvent`（数值变化）
- `ItemAcquiredEvent`（获得物品）
- `ReputationChangedEvent`（声望变化）
- `NPCMemoryAddedEvent`（NPC记忆添加）

### 替代 BigScam 硬编码

现有的 `MarketFluctuationManager` 中约500行硬编码的 BigScam 逻辑，可以用一个 JSONL 配置文件替代，实现相同的三结局分支效果。

## 🎮 玩家体验设计

### 隐藏任务的实现

- `Title` 为空 + `VisibleToPlayer: false` = 任务面板完全不显示
- 触发条件是组合的（持有特定物品 + 在特定地点 + 特定时间）
- 玩家必须自己把线索拼起来
- 只有当弧进入某个"揭示"阶段时，才通过 `notification` 让玩家意识到发生了什么

### "不作为"的后果

- 每个阶段可以设置 `timeout` 转换
- 超时后自动走默认路线（通常是坏结局）
- 例如：7天不回宗参战 → 宗门覆灭
- 这让玩家感受到"世界在运转，不会等你"

### 连锁反应

- `start_arc` 动作可以在一个弧的结局中启动另一个弧
- 例如：揭穿骗局 → 60天后触发报复弧 → 报复弧中如果有NPC记忆 → 走不同路线
- 理论上可以无限连锁，形成"蝴蝶效应"

### NPC记忆系统

- 每个NPC有一个记忆标签列表（字符串集合）
- 记忆通过 `add_npc_memory` 动作添加
- 后续弧通过 `npc_memory` / `npc_memory_not` 条件检查
- 实现"NPC记住你做过的事，并在未来做出不同反应"

## 📊 与"通马桶任务"的对比

| 维度 | 通马桶任务 | 叙事弧引擎 |
|------|-----------|-----------|
| 触发 | "去找NPC接任务" | 世界状态自然触发，玩家可能不知道 |
| 流程 | 杀10只→采5株→送信 | 有向图，任意分支 |
| 选择 | 无选择，只有做/不做 | 多选项，每个有不同后果 |
| 后果 | 奖励物品/经验 | 永久改变世界（门派覆灭/NPC死亡） |
| NPC | 无记忆的任务发布机 | 记住你的每个行为，影响未来 |
| 不作为 | 无后果 | 超时=选择，有默认结局 |
| 连锁 | 无 | 一个故事的结局触发另一个故事 |
| 重玩性 | 低 | 极高（不同选择→完全不同的世界） |

## 🔧 技术实现要点

### 文件清单（预计新增）

| 文件 | 职责 | 预计行数 |
|------|------|---------|
| `WorldStateTracker.cs` | 世界状态追踪器（标记/数值/NPC记忆/时间戳） | ~200行 |
| `ConditionEvaluator.cs` | 条件评估器（桥接所有系统） | ~250行 |
| `ActionExecutor.cs` | 动作执行器（调用所有系统API） | ~200行 |
| `NarrativeArcEngine.cs` | 核心引擎（状态机+生命周期管理） | ~400行 |
| `NarrativeTypes.cs` | 数据类型定义（弧/阶段/条件/动作） | ~100行 |
| `NarrativeEvents.cs` | 事件定义（弧激活/完成/阶段转换） | ~50行 |
| `Resources/NarrativeArcs.jsonl` | 配置文件 | 按需增长 |

### 性能考虑

- 触发检测只在事件发生时执行（不是每帧轮询）
- 休眠弧的条件检查是轻量的（大部分是字典查找）
- 活跃弧数量通常 < 10，转换检查开销可忽略
- JSONL 格式支持流式加载，不需要一次性解析整个文件

### 扩展性

- `ConditionEvaluator.RegisterCustomEvaluator()` — 注册自定义条件
- `ActionExecutor.RegisterCustomExecutor()` — 注册自定义动作
- 具体游戏可以扩展任意条件/动作类型，无需修改引擎代码
- 新系统接入只需：① 定义新条件类型 ② 定义新动作类型 ③ 注册
